## Context

v3 已把采集链路改成 API-only，并通过脚本单测和真实客户端验证了核心数据采集能力。当前剩余问题集中在 Skill 使用体验：用户第一次触发时缺少温和边界说明，歌单选择阶段会把内部 JSON 字段暴露给用户，最终 prompt 的“适合场景”说明容易被 agent 概括，reference 文档也没有在流程中形成足够清晰的路由。

v4 不改变采集事实来源、输出目录或开发期数据存放方式；它把用户可见交互和内部采集材料分层，让用户只面对必要选择，让 agent 仍保留足够诊断材料。

## Goals / Non-Goals

**Goals:**

- 为 Skill 首次回应建立固定、短句、低突兀感的用户说明。
- 在任何采集、列歌单或旧 run 读取前先检查 Python 环境。
- 将歌单列表的用户公开视图限制为编号、歌单名和曲数。
- 用固定、轻一点的主歌单选择话术解释为什么本轮只确认一个主歌单。
- 支持用户按编号或名称确认主歌单。
- 删除 `--playlist-id` 用户接口，避免把内部平台 ID 变成用户决策材料。
- 确保两版分析 prompt 的适用场景说明完整输出，不被压缩成短句。
- 采集完成后用 run 目录和文件夹用途概括收口，避免在 prompt 前堆出大量路径和 CSV 字段释义。
- 在 `SKILL.md` 中明确各 reference 的读取时机。

**Non-Goals:**

- 不处理或迁移 legacy run。
- 不移动 `outputs/` 目录，也不改变开发期用户数据存放位置。
- 不新增行为级 eval；本阶段继续使用人工测试和脚本单测。
- 不隐藏检测到的 `cloudmusic.exe` PID 或 exe 路径。
- 不改变 API-only 采集事实来源、result/csv/raw/aggregate/log 输出 schema。

## Decisions

### 用户沟通契约前置

`SKILL.md` 在标准流程前新增“用户沟通契约”。Skill 被调用后的第一条用户可见回复固定为：

```text
我会帮你采集供 AI 做听歌画像分析的数据：包括你创建的歌单列表、你确认的主歌单、最近一周和所有时间听歌排行，并整理成数据文件和分析 prompt；本轮先不直接做画像结论。
```

选择这个固定句式，是因为它同时覆盖顺利完成、环境阻塞、需要选择歌单三种路径，不会承诺下一步一定是什么命令，也不会让用户误以为本轮会直接输出画像结论。

替代方案：只在失败或选择歌单时解释 Skill 作用。拒绝原因是首次触发时最容易突兀，后置解释不能解决用户对采集范围和边界的第一印象。

### 环境检查是前置步骤

`SKILL.md` 新增精简“Python 环境”小节，要求开始采集前必须先运行：

```powershell
python scripts/collect_ncm_profile.py --check
```

并记录脚本需要 `Python 3.10+`、`psutil`、`requests`、`websocket-client`。这让 agent 先确认运行环境，而不是在缺包报错后再补救。

替代方案：继续让脚本命令失败后提示安装。拒绝原因是失败后修复会打断用户流程，也会让首次体验显得粗糙。

### 公开歌单视图和内部采集材料分离

`--list-playlists` 仍可在内部保留完整解析所需字段，但对外公开选择视图只包含：

```text
index
name
trackCount
```

用户可见表格固定为“编号 / 歌单名 / 曲数”。`playlistId`、`playCount`、`specialType`、`privacy`、`updateTime`、`source` 等字段不得进入用户选择表格。内部采集仍通过 API 获得真实歌单 ID，用于调用 `/api/v6/playlist/detail`。

替代方案：保留 `playlistId` 作为备用选择列。拒绝原因是用户心智只需要编号和名称，平台 ID 会增加决策噪声，并且截图测试已经证明 agent 会把它原样暴露。

### 主歌单选择支持编号和名称，不支持 playlistId

新增 `--playlist-index`，保留 `--playlist-name`，删除 `--playlist-id`。用户优先按编号选择；当用户明确给出歌单名时，仍先列出歌单，再让用户确认唯一主歌单。

选择“编号 + 名称”的组合，是因为它贴合用户心智：编号适合刚看完列表直接选择，名称适合用户已经知道目标歌单。删除 `playlistId` 可以避免公开接口和用户表格继续围绕内部 ID 展开。

替代方案：保留 `--playlist-id` 但不展示。拒绝原因是隐藏但仍存在的用户接口会继续诱导 agent 在歧义时暴露 ID，不如直接删除。

### 主歌单选择话术固定但不生硬

列出歌单后，用户可见说明固定为：

```text
下面是你创建的歌单。主歌单会作为这次画像数据的对照基准(只能选一个喔www)。你可以回复编号或歌单名。
```

这句话把“只能选一个”的限制放在括号里，用更轻的语气承接，同时在前半句说明理由：主歌单是这次画像数据的对照基准。这样用户不会觉得被命令，也能理解为什么本轮不采多个主歌单。

替代方案：写成“请选择一个且只能选择一个编号作为主歌单”。拒绝原因是语气生硬，而且没有说明单主歌单约束背后的数据结构理由。

### 成功输出只展示 run 目录和文件夹用途

采集成功后的用户可见收口固定为：先说“采集完成，本轮只整理数据和 prompt，不直接输出画像结论”，然后只展示本次 `run_dir`，再概括里面几个文件夹的用途：

```text
aggregate/aggregate.json
result/*.jsonl
csv/*.csv
raw/*.jsonl
log/collection_diagnostics.json
```

不要在 prompt 前逐条列出每个 result、csv、log 文件的完整路径，也不要展开 CSV 字段释义。CSV 字段释义仍可以留在 `SKILL.md` 文档里，但采集完成回复只提示：“CSV 字段释义这次不展开；如果你看 CSV 时有不清楚的字段，可以直接问 AI。”

Prompt 本身仍保留完整文件路径，因为 prompt 要可复制可用；变化只是把 prompt 前的路径清单收掉。

替代方案：继续在成功回复中列出所有文件路径和 CSV 字段释义。拒绝原因是真实采集测试显示这会让用户在进入 prompt 前看到大量路径和字段，注意力被工具细节吃掉。

### Prompt 适用场景是输出资产

两版 prompt 后的“适合场景”说明必须使用 `SKILL.md` 原文完整输出。它们不是普通解释性文字，而是帮助用户选择 prompt 风格的产品文案。

替代方案：允许 agent 按上下文改写。拒绝原因是人工测试已经看到改写会把温度和细节压成一句泛泛总结。

### Reference 路由放进流程

`SKILL.md` 在相关阶段明确 reference 的读取时机：

- 环境检测、进程、端口、路径：`references/environment.md`
- API 结构、歌单筛选、接口字段：`references/api-patterns.md`
- result/raw/csv/aggregate/log 字段变更：`references/schemas.md`
- 失败排查、自修复：`references/troubleshooting.md`

这样 agent 不需要等到文末才知道去哪找细节，也能减少漏读或一次性加载全部 reference 的概率。

## Risks / Trade-offs

- 删除 `--playlist-id` 可能让内部排错少一个直接入口 → 用 `--playlist-index` 和 `--playlist-name` 覆盖用户选择，真实 ID 继续由脚本内部解析。
- 公开视图过窄可能不足以区分同名同曲数歌单 → 当名称匹配多个或编号未明确时，要求用户按编号选择；必要时脚本错误中仍可给 agent 内部调试信息，但不把 ID 做成常规用户列。
- 固定首次回应可能在重复调用时显得模板化 → 契约规定只在 Skill 被调用后的第一条用户可见回复说一次，后续步骤不重复解释。
- 主歌单选择话术里的 `www` 有轻松感但不适合所有用户 → 这是用户明确指定的品牌化语气，只用于选择歌单这一处，不扩散到诊断、错误和数据说明。
- 成功输出不列所有路径可能让用户找具体文件慢一点 → 用 run 目录作为统一入口，必要时用户可追问具体文件路径；prompt 仍包含可直接使用的完整路径。
- Prompt 场景说明逐字输出会占用更多回复空间 → 这是用户明确要保留的温度和选择说明，优先级高于极短总结。
